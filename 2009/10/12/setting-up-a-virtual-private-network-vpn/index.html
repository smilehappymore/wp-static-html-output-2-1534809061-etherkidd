Setting up a Virtual Private Network

Oct 12th, 2009
Leave a comment | Trackback

Here’s a very simple guide of setting a Virtual Private Network (VPN for short) on your vpslink and tunnel traffic out.  I set this up because I have a large amount of unused bandwidth on my vps and alot of free open wireless networks. This was the best way to secure my wireless connection without any additional costs. The process is an extremely simple one.  In fact you should be able to set one up quite easily within 15 minutes.  Here’s a guide to do thist on Centos 5.3 32bit xen platform.   However u can simply copy and paste the server.conf for openvpn and the client config file to work after building.  For simplicity sakes we will not go through actually compiling it but using the yum tool in centos.  The first thing you should have done is set up the DAG repository at http://dag.wieers.com/rpm/FAQ.php#B. Basically CENTOS 5.3 is Redhat Enterprise linux 5.  32bit is the i386 and 64bit is x86_64.  This makes installing most packages for CENTOS much easier.  So for me it was:

rpm -Uhv http://apt.sw.be/redhat/el5/en/i386/rpmforge/RPMS/rpmforge-release-0.3.6-1.el5.rf.i386.rpm

Now issue the following command when the repository is installed:

yum install openvpn

This installs

openvpn              i386              2.0.9-1.el5.rf

and

dependencies:
lzo2                 i386              2.02-3.el5.rf

Pretty good.  Everything is installed and all you need to do is configure the openvpn server on the VPS and your client (In this case a windows 7 o/s on my computer)

cp -r /usr/share/doc/openvpn-2.0.9/easy-rsa /etc/openvpn/certs
cd /etc/openvpn/certs
<edit vars file- last five lines>
. ./vars
<gets notice: NOTE: when you run ./clean-all, I will be doing a rm -rf on /etc/openvpn/certs/keys>
./clean-all
<gets notice: -bash: ./clean-all: Permission denied>
chmod +rwx *
./clean-all

./build-ca

./build-key-server server
<Make sure to set organization: example IT and commonnam: example domain.com than sign y/y>
<ok not to set password>

./build-key client1
<Make sure to set organization: example IT and commonnam: example domain.com than sign y/y>
<ok not to set password>
./build-dh
cp /usr/share/doc/openvpn-2.0.9/sample-config-files/server.conf /etc/openvpn
cp keys/ca.crt ../
cp keys/dh1024.pem ../
cp keys/server.key ../
cp keys/server.crt ../
cd /etc/openvpn/

<problems seem to be you may need some files in the /etc/openvpn even if config has it>
cp server.csr to /etc/openvpn
cp ca.key in /etc/openvpn
#notes. proto udp doesn’t seem to work
#notes. for windows need to add two lines of route-method exe route-delay 2
#Issue this iptables to route all traffic through the vps on the command shell

iptables -t nat -A POSTROUTING -s 10.30.0.0/24 -o eth0 -j MASQUERADE

Save the above rule to /etc/rc.local to work after reboot
#routing does not save iptables after setting forward in
echo “1” > /proc/sys/net/ipv4/ip_forward
Edit /etc/sysctl.conf
net.ipv4.ip_forward = 1
#for http://openvpn.net/index.php/open-source/documentation/howto.html#mitm
#WARNING: No server certificate verification method has been enabled.
#set in client this flag
#ns-cert-type server

—————————————————-
Here’s a working openvpn server.conf file for the lazy:

local 127.0.0.1 #you need to change this to the ip of your server
port 10011 #default is 1194. before 2.0b16 was 5000
#For an example.  Make this unique but check http://www.iana.org/assignments/port-numbers
proto tcp-server
dev tun
ca  ca.crt              #  path to the CA.crt file
cert server.crt           #  path to the server certificate
key  server.key         #  path to the server key file
dh  dh1024.pem       #  path to the generated D-H parameters
duplicate-cn
server 10.30.0.0 255.255.255.0 #we issue an iptables command above for this line
client-to-client
push “dhcp-option DNS 4.2.2.2” #verizon free dns server but you can set whichever you want to client
push “redirect-gateway def1”
keepalive 10 120
comp-lzo
persist-key
persist-tun
user nobody
group nobody
max-clients 3
log vpn.log
#consider set verb(ous) to a lower setting as it creates a large log file
verb 5
chroot /tmp

————————————————————
Install installation package at http://openvpn.se/download.html
Make sure to move the ca.crt,ca.key and the “./build-key (.crt,.csr,.key)”
Here’s a working openvpn client.ovpn file for the lazy:

client
dev tun
proto tcp-client
remote 127.0.0.1 #this is the ip of your vps
port 10011 #For an example.  Make this unique but check http://www.iana.org/assignments/port-numbers
float
resolv-retry infinite
nobind
persist-key
persist-tun
ca ca.crt
cert client1.crt
key client1.key
ns-cert-type server
comp-lzo
verb 5
ping 15
ping-restart 45
ping-timer-rem
tun-mtu 1500
tun-mtu-extra 32
mssfix 1450
##fragment can only be used in udp
#fragment 1450
route-method exe #this is needed in windows vista/7 in admin mode to set correct routing to vps
route-delay 2 #this is needed in windows vista/7 in admin mode to set correct routing to vps

##################
finished. However you should be sure to set a secret. This is merely to get your openvpn working and routing your traffic through your vps giving you the ip of your vps server. You can check this by an internet search of what’s my ip and visiting a few sites to verify.

Some Notes:
openvpn requires udev installed and load the tun module however udev can be a problem in some systems especially openvps.  When you receive this error should you not have udev installed:

Note: Cannot open TUN/TAP dev /dev/net/tun: No such file or directory (errno=2)

manually enable it issuing command ‘modprobe -a tun‘; enable it at boot with command  ‘echo “tun” >> /etc/modules‘.  Disable udev if you wish to do this by editing out ‘/sbin/start_udev’ in ‘etc/rc.d/rc.sysint’ .

If you have a firewall such as csf or apf.  You have to apply the rule

iptables -t nat -A POSTROUTING -s 10.30.0.0/24 -o eth0 -j MASQUERADE

in the postrule section of the firewall configuration and not for example rc.local as it will be overwritten by the firewall. If you have openvz you may need to use snat like this with “container.ip” as your vps ip

iptables -t nat -A POSTROUTING -o venet0 -j SNAT --to-source <yourvpsip>

Caveat emptor

Ipv6 is quickly becoming a requirement.  With Ipv4 most likely used up by 2012.  It is probably best to disregard the settings to disable ipv4 in this guide and use a later  version of openvpn 2.1.1 (dec 11, 2009).

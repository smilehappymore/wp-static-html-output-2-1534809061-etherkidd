Setting Exim4 on Low End VPS with gmail

Oct 10th, 2012
No comments

In this article I will show you how I setup Exim4 on Debian 6 on a low end vps. In general if your vps provider has a minimal template (5-10M memory usage); I use that and follow the below steps to setup gmail. You would basically want to use this configuration if you don’t plan to have an email server on your vps. Since i Usually set this up on a 64mb xen (with 128mb swap) or a 128mb openvz (with 256mb burst); I don’t intend for the vps to have to send out alot of mail. Gmail accounts allow up to 500 pieces of mail per day so any active forum would NOT make this a good choice but considering the memory usage from database and cpu load of visitors… choose a vps with more memory.   I get around 10-20 pieces of mail for server notifications like cron, etc. while having your website(wordpress) use outgoing smtp configuration of the same gmail account for other light email exchanges. Add sudo command if you need superuser access.

aptitude install exim4 -y

If installed successfully use the following to remove sendmail or when exim4 is broken.
aptitude purge sendmail -y
aptitude install exim4-base exim4-config -y
aptitude install exim4-daemon-light -y
aptitude install exim4 -y

While you have installed exim4. It’s possibly broken in a minimum template and you need to set it up properly. For a quick and dirty way I use the following:

aptitude install exim4-config -y

It is important now to read exim4 configurations if you want a good understanding on why I choose the following selections after I start the “dpkg-reconfigure exim4-config” .

dpkg-reconfigure exim4-config
————-<these are my selections>————–

  • type of mail configuration – mail sent by smarthost; no local mail
  • mail name – localhost
  • ip listen – 127.0.0.1
  • other distinations – <blank>
  • visible domains – localhost
  • outgoing smarthost – smtp.gmail.com::587
  • DNS queries minimum – no
  • split configuration – no
  • root and postmaster recipient – <blank>

Now you are almost done.  You just need to edit /etc/exim4/passwd.client and add the following lines with <user> and <password> with your gmail login:

gmail-smtp.l.google.com:<user>@gmail.com:<password>
*.google.com:<user>@gmail.com:<password>
smtp.gmail.com:<user>@gmail.com:<password>

Almost done.  now update configuration:

update-exim4.conf

The last step is optional…  While this is setup to work you still need to setup a root alias otherwise unconfigured system messages to root will throw undeliverable messages into your gmail account.  If you find it a little annoying to see those messages than edit /etc/aliases with the following but replacing <user> with your gmail name:

root:                                   <user>@gmail.com

Run the update so the alises are used immediatelly:

newaliases

Finally test your mail is working

mail -s “test mail” root </dev/null

In conclusion this is a perfect solution for mail that doesn’t have a fully qualified domain name (fqdn); this enables mail to end up in someones mailbox far more frequently than using your own domain without setting up a reverse dns for your mail server and appropriate spf record.  The obvious downside is it’s a gmail account with mail sending limitations.  You can however use google apps for your domain but I generally don’t when setting up a low end vps.  If you just want to setup exim to deliver mail without gmail than you can edit /etc/exim/update-exim4.conf.conf and change the line dc_eximconfig_configtype=’local’ to dc_eximconfig_configtype=’internet’ and restart exim4 with /etc/init.d/exim4 restart

Hope this is helpful to someone and any additional tips, tricks, comments you want to add that would be great.

ezpzhosting.co.uk budget vps review

Jan 30th, 2012
No comments

So I have not been happy with ezpzhosting.co.uk reseller hosting for a while; a recent uptime notification spam in January made me request a credit so I can use some other service that I might be happy with.  Fortunately Dan at ezpzhosting.co.uk was understanding and allow that to happen.  So this is my review which is somewhat favorable especially since the performance of my wordpress on the vps is much more responsive.  So here’s a quick list of the pros and cons of my budget vps package in the united kingdoms in the city of Kent (datacenter) named u-vps-128. 128mb ram with burst up to 256mb:

Pros:

  • 500gb bandwidth.  very generous
  • £4.99 equals to around $7.65 us dollars.  This will fluctuate higher or lower with currency.  A good price considering the bandwidth.
  • There are discounts for prepaying ahead of 6 months and a year but has a no refund policy; so be cautious and use your own judgement.
  • SolusVM control panel management.  has serial console so you can access it even if ssh is not installed or failed.

Cons:

  • only openvz available.  xen would be preferable
  • Uses older kernel. uname produces 2.6.18-238.9.1.el5.028stab089.1 in debian
  • no ipv6 available
  • reverse dns requires a ticket and currently disabled in soluxvm
  • Access to only 1 cpu core at 2327.496hz. Actually not a bad thing.

Troubleshooting: I did run into several problems with my vps which leads me to believe they are lacking a bit of experience in the field; I’ll detail some of the issues I’ve experienced in no particular order.  First thing to get out of the way is, I seemed to had purchased the vps when they were planning to move their datacenter.  Originally the vps was in Manchester and it has now moved to Kent.  This is not a big deal but this usually only happens to smaller vps providers.  The move caused an ip to be issued that was different than the one I had, but data remained the same.  The second issue was the solusvm control panel showed a package bandwidth limitset at 100gb instead of the 500gb advertised.  This was quickly fixed with a support ticket.  The third issue is the vps templates was not tested properly.  For example debian 5 did not have locales installed so I could not install the linux benchmark program properly.  The iptables modules like recent was not working so I could not secure the vps a little.  In debian 5 the default sources list was not correct so I had to edit the file to do an update.  Aptitude was not installed in debian 5 so that had to be installed.  Updates was still not working because the default templates had an init file “/etc/init.d/vzquota” which did not conform to standard debian init.d script requirement.  The file /etc/init.d/vzquota will also get rewritten back to default on reboot so an init.d script had to be written to append the “/etc/init.d/vzquota” to correct the issue.  Everything however is now working properly and I’m happy with the vps.  This review does not include any network or hardware issues as I’ve not encountered any yet however I will update it as appropriate, when it happens.  So far though I would recommend the vps if you are looking for large bandwidth in the United Kingdom as the price is very competitive.

Recommended:  I use debian squeeze because it is easier to configure for a lower memory footprint.  However ezpzhosting.co.uk’s debian 6 squeeze minimal template includes rsyslogd which allocates a large amount of memory in privvmpages on an openvz system.  This caused my mysql to partially fail installation so I recommend you remove it and install something like inetutils-syslogd.  However the vps uses an old kernel that is more suited for Centos.  You will probably experience fewer problems using Centos.

Incapsula vs Cloudflare – Free Version

Oct 10th, 2011
No comments

I felt a little bored and from a comment on this blog…  I decided to see what Incapsula is all about.  From some internet searches… it appears to be get random reviews in some large forums from new users.  I instantly thought these were paid reviews as  they are very suspicious and I tried not to take too much weight into them.  From looking at the features from Incapsula compared to cloudflare;  it did not appear to be anything worth wasting too much time on simply because of the pricing.  Since I already covered Cloudflare in my previous post I’ll just detail the differences rather than being repetitive.  This is a comparison of the free version of incapsula (allowing 30 days to pass for their free trial gold plan) and free cloudflare.

Sign up:
The sign up process was as easy as cloudflare and very similar.  Where cloudflare will function as a dns replacement… incapsula will actually work well with your existing dns provider or your own name servers.  This of course has the benefit of eliminating one aspect of failure which any experienced web professional will tell you is quite important.  Managing a small to medium reliable dns nameserver is not hard and worth learning.  Unfortunately at incapsula they require an a record for your main domain and for any subsequent subdomain like www;  you have to use a cname.  A cname dns lookup will cause a slightly longer delay but should not be an issue if your name servers are reliable which is the first key thing to remember in distinguishing between the two.

Setup:
Incapsula decides to start your threat rules to “observe and report”.  The only other option is block request and do nothing.  This has the benefit of acting simply as a content delivery network up to 25gb monthly transfer limit.  You can add more domains but the cname is different per domain.  Unfortunately the 25gb limit does not appear to be per domain  as it only shows an accumulated bandwidth for the last 30 days on the “Account tab” which is more than what’s listed when adding the domains on the “My Sites” tab.  Someone correct me if I am wrong.

Speeds:
Because my site is small it is generally hard to see which service is faster.  Furthermore Incapsula does not really provide much info about their datacenters.  Since speeds really depend greatly on the distance of your webserver with the cdn datacenter due to it’s continuous connection requirements.  Then there is the client computer which will have to connect to the cdn datacenter which makes any test unreliable.  However I will say it is faster with both services vs not using them; but you can find a variety of speed tests that you can absorb to your hearts content and make your own judgement.  However is is important to be aware of the ip space of both providers or any content provider you have to prevent any firewall issues.

Personally I find cloudflare to be faster and has a much larger ip space.  In addition to having a better idea of their datacenters you can setup a vps that should ideally be closer but it also worth having good knowledge of the particular geographic location of your target audience.  Incapsula only has a standard “acceleration mode” option for free accounts while cloudflare has some features like minify (in beta).  Nginx supports set_real_ip_from with ipv6 starting with versions 1.3.0 and 1.2.1.  For .htaccess and nginx rules and a list of when they are updated go to cloudflare and incapsula.

Protection and Features
Unfortunately I do not have expert knowledge in threats which I can test the network against.  I would suggest doing an internet search but the threat opinions of the authors can be quite suspect since it does depend on alot of things.  Since the default of Incapsula is to report and do nothing… Cloudflare is on medium by default… I would tend to think cloudflare is more secure. 

Incapsula has “access rights” which you can block access by regions and client applications by simply clicking on their list.  By selecting these options it can give you greater security but also a great big headache as ip ranges can change which changed 09/20/11 that removed a list of selections for geographic and client applications; you now can type in that info. For example; alot of ip ranges can change geographically because web hosting providers with a large pool decided to have servers in a foreign country.  This probably would not affect normal fios/dsl/cable ip ranges from your local isp like comcast but probably impact server  ip ranges from web host providers.  By far the best feature is the email notification.  You really do not know how many times google has managed to kill one of my sites so it is nice to get notified when they attack.  Unfortunately incapsula does not have a whitelist ip range available but probably not necessary.

Cloudflare while lacking email notification which is something I love, has a greater “threat control” options by listing threats and ips you can block or trust.  Unfortunately it does lack incapsula’s client application list. In addition to a more control over ip trust/deny; they have cloudflare apps.  These apps are most useful to those who have google analytics, webmaster tools, and pingdom while giving some misc apps like cdnjs a worthy look.  The beta options like minify and rocketloader will also show a boost on heavy websites but can spell trouble if unnecessarily turning them on.

Support and Documentation
Major issue with Incapsula is the lack of documentation.  You basically have to log into Incapsula to even get a list of their ip ranges which is useful to know you want to make sure they are whitelisted if you have a firewall so your entire website is blocked to the outside world.  Even frustrating is finding this info which they titled “Restricting direct access to your website”.  Cloudflare has shown to be very heavily documented though information is rather scattered and certain topics like their ip ranges should be centralized so any update will be reflected on multiple threads on how to set that up.  Cloudflare support is exceptional in my opinion and you can get an answer very quickly vs Incapsula which requires you to create an account to look at their support forum and sparse documentation with a hopeful 8 “business” hour response time.  You really get to feel like a customer at cloudflare before actually paying.

Conclusion
Incapsula will have it uses.  The limited 25gb bandwidth account is not a huge problem and not that difficult to get around. One easy way is redirect all js/images/css to coral network via a simple redirect or you can use a commercial cdn.  However it is clear the service is geared as a service firewall with some light cdn.  So if you are already paying or managing your separate but reliable dns, mail,  internet application, storage, and other internet service… it’s not a much larger cost to include a firewall.  The most important thing is to make sure the server using incapsula is completely anonymous like having outgoing mail revealing it’s ip address or sharing it with some service.  The stable infrastructure and simple features serves it’s intended purpose of securing your website as long as you enable them.

Overall I would choose cloudflare but it’s popularity is mainly the issue.  There is a more than a frequent chance someone else using cloudflare will cause a ddos and impact you by association.  However if you have a site consuming large amounts of bandwidth that solicits bots which spam you with comments, than cloudflare has a better track record and experience.  I find there are more blips in uptime and stability compared to incapsula but should perform faster overall while reducing alot of spam and maintenance time.  The bright side is currently cloudflare has an automatic ipv6 beta and since they handle the dns…  this is one step you can avoid with incapsula when they support it; though even my vps provider have issues with ipv6 still.

Both services have their hiccups and minor issues.  Incapsula dashboard stated my site was not responsive and I would get a notification when it changes but that did not happen. Incapsula updated their dashboard on 09/20/2011 which also caused all my sites for over 3 hours to be seen with their 404 error enclosed in google search even as my logs state it’s accessing my server just fine as “Incapsula Uptime Monitor”.  Stats can be somewhat deceiving and these services will change over time with new features or existing features changed.  Always make a backup plan should you choose to discontinue their service.

BottomLine
Get a reliable host for great uptime.   Adding another service in front of your hosting is just another layer of possible failure and headache.  You also give up some control when using these services like the inability to fully customize the challenge/block/404 pages, etc. You can experiment with torbit, yotta, etc.  See which one works for you.  Just prepared for downtime as I have the worst time with these experiments and if your sites is large enough, an hour plus downtime is not acceptable.

Interesting but obvious fun thing is you can use cloudflare with many of it’s competitors which only require a dns record change.  Obviously that can cause some serious troubleshooting problems when things go wrong but it worked when I was trying it out.
